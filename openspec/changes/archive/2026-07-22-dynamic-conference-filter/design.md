## Context

`KanbanBoard.tsx` хранит `selectedSectionId` с начальным значением `TECHLEAD_SECTION_ID = 10634778` и при монтировании вызывает `load([TECHLEAD_SECTION_ID])`. Клиент передаёт эти id в `filters.sections`, а `filters.conferences` всегда остаётся пустым.

Актуальные метаданные содержат три конференции и 30 секций, секции `10634778` среди них нет — она принадлежала Saint TeamLead Conf 2026. Шапка рисует все 30 секций одним плоским `<select>`, где названия вроде «Другое» или «AI-трансформация…» неотличимы по принадлежности.

Ключевое наблюдение: `get_info` уже возвращает ровно те конференции, к которым у пользователя есть доступ. Отдельного эндпоинта прав не нужно — «доступные конференции» это и есть `result.conferences`.

## Goals / Non-Goals

**Goals:**
- Выбор конференции строится из фактически доступных пользователю данных, без захардкоженных id.
- Список секций всегда даёт понять, к какой конференции секция относится.
- Выбор переживает закрытие доски.
- Устойчивость к исчезновению конференции или секции из метаданных (ровно тот случай, который сломал текущий дефолт).

**Non-Goals:**
- Мультивыбор конференций или секций — оба селектора одиночные.
- Фильтрация панели кураторов по `curator.conferences` из метаданных: панель по-прежнему строится из загруженных заявок и самоподстраивается.
- Кеширование заявок по конференциям — при каждой смене выбора идёт полная перезагрузка.

## Decisions

### Два зависимых `<select>` вместо одного списка

Шапка получает `[Конференция ▾] [Секция ▾]`. Второй селектор фильтруется по первому:

```
selectedConferenceId = 10560392        selectedConferenceId = null
        ↓                                      ↓
[Секция ▾]                             [Секция ▾]
 • Все секции                           • Все секции
 • Архитектура и масштабируемость        ┌ AI Native Conf 2026 ────
 • Базы данных и системы хранения        │ • Агентный инжиниринг
 • ... (19 секций HighLoad++)            │ • ...
                                         ┌ HighLoad++ 2026 ────────
                                         │ • ...
```

`<optgroup>` появляется только в режиме «Все конференции». При выбранной конференции группировка избыточна — все секции и так из одной, а списки короткие (4 / 19 / 7).

Альтернатива — один селектор с `<optgroup>` всегда — отвергнута: она не даёт выбрать конференцию целиком, а именно этот режим нужен чаще всего.

### Фильтр строится без спецслучаев

```
conferences: selectedConferenceId ? [selectedConferenceId] : []
sections:    selectedSectionId    ? [selectedSectionId]    : []
```

Если в режиме «Все конференции» выбрана конкретная секция, уходит `conferences: [], sections: [id]` — секция сама сужает выборку до своей конференции. Первый селектор при этом **не** переключается автоматически: пользователь видит ровно то, что выбрал.

`fetchAllLectures` меняет сигнатуру на `fetchAllLectures(filters: { conferenceIds: number[]; sectionIds: number[] })` — именованный объект вместо двух позиционных массивов чисел, которые легко перепутать местами.

### Дефолт и восстановление выбора

Приоритет при инициализации:

```
chrome.storage.local: { conferenceId, sectionId }
        │
        ├─ conferenceId есть в meta.conferences?  ── нет ──┐
        │                  да                              │
        ├─ sectionId есть в meta.conference_sections       │
        │  и принадлежит conferenceId?                     │
        │      нет → sectionId = null                      │
        ▼                                                  ▼
   восстановленный выбор              meta.conferences[0], sectionId = null
```

Сохранённое значение `conferenceId: null` («Все конференции») валидно и восстанавливается как есть — отличать «не выбрано» от «отсутствует в storage» нужно по наличию ключа, а не по `null`.

Валидация обязательна: именно её отсутствие привело к текущему состоянию, когда дефолт указывает на исчезнувшую секцию.

Хранение выносится в отдельный модуль `src/storage/preferences.ts` с двумя функциями (`loadPreferences`, `savePreferences`) — компонент не должен знать про `chrome.storage`, а сам модуль тестируем с подменой `chrome`.

### Порядок инициализации

Сейчас `useEffect` при монтировании грузит метаданные и заявки одним `load()`. С восстановлением выбора появляется зависимость: набор фильтров известен только после получения метаданных. Загрузка разделяется на два шага:

```
mount → fetchGetInfo() ─→ meta
                          │
                          ├→ loadPreferences() + валидация → выбор
                          │
                          └→ fetchAllLectures(выбор) → заявки
```

Экран загрузки покрывает оба шага, состояние `loadState` остаётся `loading` до конца.

### Разрешение `storage`

`public/manifest.json` сейчас объявляет только `tabs`. Для `chrome.storage.local` добавляется `storage`. Разрешение не требует подтверждения пользователем при обновлении расширения.

## Risks / Trade-offs

- **Полная перезагрузка заявок при каждой смене конференции** — на HighLoad++ с 19 секциями страниц может быть много, ожидание заметное. → Экран загрузки уже есть; кеширование по ключу фильтра сознательно отложено до появления реальной жалобы.
- **Сохранённая секция принадлежит другой конференции, чем сохранённая конференция** (рассинхрон после ручной правки storage или смены метаданных) → Валидация проверяет принадлежность секции конференции и сбрасывает секцию, а не весь выбор.
- **`chrome.storage.local` недоступен или бросает** (нестандартная сборка, отозванное разрешение) → `loadPreferences` возвращает пустой результат, `savePreferences` глушит ошибку; доска работает с дефолтом, а не падает.
- **Смена конференции при активном фильтре кураторов** оставила бы выбранными id из чужой конференции → Смена конференции сбрасывает `selectedCuratorIds` (поведение уже есть в `handleSectionChange`, распространяется на конференцию).

## Migration Plan

Разовое изменение без миграции данных: до первого сохранения storage пуст, срабатывает ветка дефолта. Откат — возврат предыдущей сборки; осиротевший ключ в `chrome.storage.local` старой версией игнорируется.
